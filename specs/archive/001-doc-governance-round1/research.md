# 研究记录：Round 1 项目文档结构整改

## 决策 1：建立项目级 arc42 骨架

- **决策**：在 `docs/architecture/` 下建立项目级最小 arc42 骨架，章节集采用澄清阶段确认的结果。
- **理由**：当前仓库的项目级真相分散在 `SPEC.md`、`PROJECT_BRIEF` 和执行类规则文档中，
  但没有明确的长期架构落位。
- **备选方案**：
  - 继续只用 `SPEC.md`：拒绝，因为它混合了范围、协议和 taxonomy，缺少分层架构视角。
  - 把所有内容塞进一个大治理文档：拒绝，因为只会在新位置复制旧的混杂问题。

## 决策 2：模块级 arc42 骨架只在重点审核模块建立

- **决策**：Round 1 只为 T04、T05-V2、T06 创建模块级架构骨架。
- **理由**：这三个模块是当前混杂最严重、人工审核优先级最高的对象。
- **备选方案**：
  - 一次性为所有模块建骨架：拒绝，因为会扩大范围，但不会提升本轮审核质量。
  - 本轮完全不建模块骨架：拒绝，因为 Round 2 迁移时将缺少可信落点。

## 决策 3：T05-V2 保持物理独立，同时文档上归入 T05 family

- **决策**：目前将 `t05_topology_between_rc_v2` 视为独立模块路径，但在治理文档和迁移映射中，
  明确标注它属于 T05 family。
- **理由**：仓库证据已经显示它拥有独立的 `modules/`、`src/`、`tests/` 和 `scripts/`，
  同时命名和业务域又明显与 legacy T05 保持连续性。
- **备选方案**：
  - 立即合并进 legacy T05 文档：拒绝，因为当前 repo 已把它当作独立可执行模块。
  - 完全当成与 T05 无关的新模块：拒绝，因为会丢失家族连续性和迁移上下文。

## 决策 4：旧文档保留，先映射后迁移

- **决策**：所有旧文档先保留原位，先建立清晰的迁移映射，再考虑后续重写或重定位。
- **理由**：这是一个 brownfield 仓库，尤其 T05 的审计和验收材料具有很高历史价值。
- **备选方案**：
  - 立即移动到新目录：拒绝，因为会制造不必要的 churn，并掩盖尚未澄清的落位问题。
  - 先删除明显重叠的旧文档：拒绝，因为现阶段尚未完成统一归口。

## 决策 5：收缩 AGENTS 和 SKILL 的职责

- **决策**：未来的 `AGENTS.md` 只保留持久执行规则和指针；未来的 `SKILL.md` 只保留可复用流程。
- **理由**：当前很多 AGENTS 和 SKILL 已经承载了稳定业务真相，导致治理边界模糊、难以审核。
- **备选方案**：
  - 保持现状不变：拒绝，因为这正是本轮文档治理要解决的核心问题。
  - 彻底删除 AGENTS 或 SKILL：拒绝，因为两者在正确收敛后仍然有明确价值。

## 决策 6：把 T03 和 T10 作为治理例外处理

- **决策**：Round 1 只把 `t03` 记录为缺失的 taxonomy 成员，把 `t10` 记录为超出原 taxonomy 且存在命名漂移的模块；
  本轮不做结构归一化。
- **理由**：这两个问题都需要在治理上显式可见，但都不属于本轮深度迁移范围。
- **备选方案**：
  - 现在就补齐缺失的 T03 结构：拒绝，因为当前仓库没有可锚定的实现。
  - 现在就重命名 T10 路径：拒绝，因为这属于比第一轮治理更大的结构变更。

## 决策 7：Round 1 不执行 root agent-context 生成

- **决策**：本轮不执行 `.specify/scripts/bash/update-agent-context.sh codex`。
- **理由**：它会在目标治理结构尚未稳定前，提前创建或更新 root 级 `AGENTS.md`。
- **备选方案**：
  - 为了流程纯度仍然执行：拒绝，因为会引入任务书未要求的长期文档面，反而增加治理歧义。
