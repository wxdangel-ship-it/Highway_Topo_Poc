# 规格说明：单文件体量约束 + 执行入口脚本治理

**变更标识**：`008-code-boundary-entrypoint-governance`  
**当前 Git 分支**：`codex/008-code-boundary-entrypoint-governance`  
**日期**：2026-03-17  
**状态**：Draft

## 1. 背景

当前仓库已经完成主阅读路径、治理基线和模块生命周期收口，但 repo 级约束里仍缺少两类边界：

1. 单文件体量的结构债约束
2. 执行入口脚本的准入与登记约束

仓库中已经存在多份超阈值源码文件，以及数量较多、来源层级不一的执行入口脚本。如果不先固化规则并给出现状基线，后续迭代容易继续膨胀。

## 2. 目标

1. 将单文件体量约束正式写入当前项目约束
2. 将执行入口脚本治理正式写入当前项目约束
3. 建立当前态的超阈值文件审计
4. 建立当前态的执行入口注册表
5. 保持 root `AGENTS.md` 简洁，只增加最小、可执行规则

## 3. 非目标

- 不修改算法、测试、运行脚本或入口逻辑
- 不拆分任何现有大文件
- 不删除任何现有脚本
- 不引入 CI / pre-commit / lint 工具
- 不进行目录重构或模块级重构

## 4. 澄清结论

### 4.1 纳入体量约束的文件类型

当前纳入约束的是纳入版本管理的源码 / 脚本文件，重点包括 `.py`、`.sh`、`.cmd`、`.ps1`、`.js`、`.ts`、`.bat` 等。

### 4.2 `100 KB` 阈值适用范围

`100 KB` 阈值只针对源码 / 脚本文件；文档、数据、导出产物和第三方依赖不属于本轮主体。

### 4.3 什么算执行入口脚本

当前把 repo 级命令包装、`scripts/` / `tools/` 下独立脚本、模块级 `__main__.py` / `run.py` 类入口、以及独立验证 / 审计脚本视为执行入口。

### 4.4 什么算新增入口的合理例外

只有当现有入口无法通过参数化、配置化或模块内复用解决，且 `SKILL.md` 也不能承接，并且任务书明确批准时，才允许新增入口。

### 4.5 规则落位

- root `AGENTS.md` 只写最小规则
- 详细解释写入 `docs/repository-metadata/code-boundaries-and-entrypoints.md`
- 现状审计分别写入 `code-size-audit.md` 与 `entrypoint-registry.md`

### 4.6 完成标准

- repo 级规则已补齐
- 超阈值文件审计已形成
- 执行入口注册表已形成
- 本轮没有引入新的治理冲突

## 5. 成功判定

1. 维护者阅读 `AGENTS.md` 即可知道两条新规则
2. 阅读 `docs/repository-metadata/README.md` 即可找到详细约束、体量审计和入口注册表
3. 后续若继续修改超阈值文件或新增入口，已经有明确的说明与登记位置
