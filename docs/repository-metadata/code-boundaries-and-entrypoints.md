# 当前代码边界与执行入口约束

## 1. 文档目的

本文档用于说明当前仓库对“单文件体量”和“执行入口脚本”的正式约束。
它描述的是当前规则，不是未来迁移蓝图，也不直接发起代码拆分或脚本收敛。

## 2. 约束范围

- 约束主体是纳入版本管理的源码与脚本文件，重点包括 `.py`、`.sh`、`.cmd`、`.ps1`、`.js`、`.ts`、`.bat` 等。
- `tests/`、`tools/` 中同类源码 / 脚本文件也纳入体量审计，因为它们同样会累积结构债。
- 文档、数据、导出产物、`outputs/`、`runs/`、`data/`、第三方依赖目录不属于本轮约束主体。

## 3. 单文件体量约束

- 单个源码 / 脚本文件超过 `100 KB` 时，视为结构债或超阈值文件。
- 已超阈值文件可以继续存在，但后续变更若必须触碰它，必须同时给出拆分计划或结构整改说明。
- 本规则只限制“继续无计划堆叠”，不要求本轮立即拆分已有大文件。
- 当前超阈值清单见 `code-size-audit.md`。

## 4. 执行入口脚本治理

### 4.1 什么算执行入口

当前把以下独立启动面视为执行入口：

- repo 级命令包装，如 `Makefile`
- `scripts/`、`tools/` 下可以直接执行的 shell / Python 命令脚本
- 模块内可直接独立启动的 `__main__.py`、`run.py` 或其它带独立入口的脚本
- 具有明显独立启动意义的验证 / 审计脚本

以下内容默认不算独立入口：

- 被其它脚本 `source` 的共享辅助脚本
- 只提供函数、不承担直接启动职责的内部模块
- 文档中仅作为示例展示的命令片段

### 4.2 当前治理规则

- 默认禁止新增新的执行入口脚本。
- 只有当现有入口无法通过参数化、配置化或模块内复用解决，且不能由已有标准 Skill 包或已登记的模块流程文档替代，并且任务书已明确批准时，才允许新增入口。
- 任何新增执行入口脚本都必须登记到 `entrypoint-registry.md`，写明用途、适用范围和与现有入口的替代关系。
- 对重复性流程，优先复用已有入口、扩展已有参数 / 配置，或补充到标准 Skill 包，而不是继续新造 `run_xxx.*` 脚本。

## 5. 合理例外

以下情况可以视为新增入口的合理例外，但仍需任务书批准与登记：

- 现有入口无法安全表达的专项审计 / 回归入口
- 仅针对已存在运行链路补充的最小包装入口
- 明确属于迁移、维护或人工核验的一次性工具入口

“临时方便”“命令太长”“想单独试一下”不构成合理例外。

## 6. 与其它文档的关系

- root `AGENTS.md` 只保留最小规则和指针，不展开解释。
- 标准 Skill 包只记录可复用流程，不替代这里的 repo 级约束；模块目录中的流程文档也不替代这里的 repo 级约束。
- 模块级 `architecture/*` 与 `INTERFACE_CONTRACT.md` 负责模块真相，不负责 repo 级入口准入。
- 当前超阈值文件清单见 `code-size-audit.md`。
- 当前入口注册表见 `entrypoint-registry.md`。

## 7. 后续维护原则

- 本轮只做规则固化与现状审计，不拆文件、不删脚本、不合并入口。
- 后续如果继续修改超阈值文件，应先说明为什么仍在该文件上修改，以及计划如何拆分。
- 后续如果确需新增入口，应先证明复用现有入口，以及标准 Skill 包或现有流程文档都不足，再进入批准与登记流程。
